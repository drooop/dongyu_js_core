Dongyu App 用户登录与 Matrix 能力架构

推荐方案：Zitadel 做主身份源，Dongyu App 做业务会话与权限，MAS 继续服务 Matrix。

结论 系统地图 登录流程 Matrix 能力 管理总线 职责边界 落地路径 关键风险
本页是设计说明，不是运行配置。它描述推荐边界：Dongyu App 不重做 MAS，也不把 Matrix 当成总权限源。

Zitadel 负责登录身份，Dongyu App 负责业务权限，Matrix 作为可授予的通讯能力。

推荐方式不是让 Dongyu App 变成新的 SSO 平台，也不是让 Matrix 决定所有权限。 Dongyu App 只需要成为自己的 OIDC client，并把登录后的用户映射到 Workspace、角色、Matrix 绑定和管理总线能力。

主身份源 Zitadel 证明用户是谁，处理账号、组织、登录策略和多因素登录。
主权限源 Dongyu App 决定用户能看什么、改什么、触发什么。
能力基础设施 Matrix/MAS 提供聊天、通知、管理总线和 Matrix token。
Zitadel 可用，入口为 sso.dongyudigital.com
MAS 可用，入口为 mas.dongyudigital.com
Matrix SSO 已开启，支持 m.login.sso 与 m.login.token
Dongyu App 应新增自己的 Zitadel OIDC 登录与业务 session

系统地图

把“身份”“业务权限”“Matrix 能力”“消息传输”拆开，后续替换任一层都不会牵动整套系统。

身份层
Zitadel

账号、组织、登录策略、MFA、OIDC issuer。

OIDC token

只证明登录身份，不直接代表 Dongyu App 权限。

用户目录

以 Zitadel subject 或 email 映射到 Dongyu App 用户。

Dongyu App 层
App session

Dongyu App 自己签发 cookie，代表当前浏览器已登录。

权限模型

Workspace、角色、资源、操作策略都由 Dongyu App 判定。

能力发放

按权限决定是否绑定 Matrix、上传文件、发管理总线消息。

Matrix 能力层
MAS

Matrix 专用认证桥，负责把 Zitadel 登录变成 Matrix 登录。

Matrix account

用户在 Matrix 的身份，例如 @user:synapse.dongyudigital.com。

Matrix token

调用 Matrix API 所需凭据，由 MAS/Synapse 负责发放。

通讯与总线层
聊天/通知

用户消息、房间、通知和文件能力走 Matrix。

管理总线

受控的系统指令通道，不等同于普通聊天权限。

Workers/MBR

只处理通过 Dongyu App 授权后的受控消息。

推荐登录流程

Dongyu App 做 OIDC client，只为自己建立 session。它不是新的全局 SSO，也不负责给 Matrix 发 token。

1
访客打开 App

可以浏览公开页面。敏感写入、上传、管理动作保持不可用或提示登录。

2
跳转 Zitadel

点击登录后，Dongyu App 发起 OIDC authorization code flow。

3
回到 callback

Dongyu App 后端校验 code、state、nonce，再读取用户身份。

4
建立 App session

写入 Dongyu App 自己的 session cookie，后续页面请求只认这个会话。

5
加载业务权限

按用户、组织、Workspace 和角色计算可见资源与可执行动作。

从 Dongyu App session 到 Matrix token

Matrix token 是一项可发放能力，不是 App 登录本身。用户可以先登录 Dongyu App，再按需绑定 Matrix。

什么时候触发

用户第一次进入聊天、房间、文件上传、Matrix Suite 或需要个人 Matrix 身份的功能时触发。

为什么推荐

复用现有 MAS 和 Matrix SSO，用户通常已经在 Zitadel 登录，第二跳可以近似无感完成。

Dongyu App session 用户已经通过 Zitadel 登录，App 知道这个人是谁。
Matrix SSO redirect 跳到 matrix.dongyudigital.com，由 MAS 接 Zitadel。
Matrix token 回调带 loginToken，后端换取 Matrix access token 并存储绑定。

适用场景

管理总线、系统通知、审计投递等不需要代表某个自然人直接发言的能力。

边界

服务端可以用受控 Matrix 身份发消息，但必须先通过 Dongyu App 权限判断，不能让前端直接拿总线凭据。

未来能力

登录后自动创建或关联 Matrix 账号，减少用户感知到的绑定步骤。

前置条件

需要明确 Matrix 用户生命周期、停用策略、账号命名、组织隔离、数据删除和审计规则。

管理总线能力如何发放

管理总线不是“有 Matrix token 就能发”。它应由 Dongyu App 授权后再转成受控消息。

先判断 App 权限

用户是否属于这个 Workspace，是否能安装 App、删除资源、触发 Worker、查看调试信息，都由 Dongyu App 判断。

再生成总线消息

通过权限判断后，后端生成标准管理总线消息，附带操作者、资源、动作和审计信息。

最后使用 Matrix 投递

Matrix 只负责传输。可以使用用户绑定身份，也可以使用受控服务身份，取决于动作是否需要以用户名义发出。

分阶段落地路径

先让 Dongyu App 拥有自己的登录态，再把 Matrix 作为能力逐步接上。

Phase 1
Zitadel OIDC 登录

新增 Dongyu App 登录、callback、session、退出；前端右上角显示访客/已登录用户菜单。

Phase 2
业务权限模型

建立用户、组织、Workspace、角色、操作权限。敏感动作统一经过后端授权。

Phase 3
Matrix 按需绑定

用户进入聊天或 Matrix 能力时，通过 Matrix SSO 绑定 Matrix account，后端安全保存 token。

Phase 4
管理总线授权网关

所有管理总线动作先过 Dongyu App 权限，再由后端生成受控总线消息并投递。

Phase 5
生命周期与审计

补齐 token 失效、用户停用、组织隔离、审计日志、账号回收和紧急封禁策略。

关键风险与设计原则

最重要的原则是：登录、业务授权、Matrix token、管理总线指令必须分层。

  • 不要重做 MAS Dongyu App 不应自己实现 Matrix 登录兼容、Matrix token 发行或 Matrix 认证桥。
  • 不要信任前端 前端只能提交意图。上传、总线、安装、删除、调试等动作必须由后端按 session 和权限判断。
  • 不要混淆 token Zitadel token 证明登录身份，Dongyu App session 证明 App 登录，Matrix token 证明可调用 Matrix。
  • 不要让 Matrix 定义业务角色 Matrix room 成员关系可以作为辅助信号，但 Workspace 管理员、资源 owner、操作权限应由 Dongyu App 管。
  • 保留替换空间 未来如果 Matrix 只作为一个 app 或者被替换，Dongyu App 的登录与权限体系仍应继续成立。
