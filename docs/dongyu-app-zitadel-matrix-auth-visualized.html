ZITADEL 不做 Dongyu 全局身份中心；跨工作区身份走 Matrix 管理总线上的短期能力。

推荐结构是三层：每个 workspace 有自己的本地认证域；Matrix 承担跨 workspace 管理总线通讯； 跨 workspace 请求传短期、带 audience、nonce、expiry、policy 的 assertion 或 capability，不转发普通登录 token。

可以保留 ZITADEL 用于官网账号、购买、组织内 SSO、workspace 内登录。
需要避免 把 ZITADEL 设计成覆盖所有 VPN、所有 workspace 的唯一身份中心。
应当新增 Matrix 管理总线上的跨 workspace assertion / capability bridge。

设计判断

把身份登录、业务权限、Matrix 认证和跨 workspace 授权拆开，后续每一层才能独立演进。

保留

ZITADEL 作为 workspace 内 SSO

ZITADEL 适合做组织内登录、外部 IdP broker、MFA、角色 claim 和账号生命周期。

收窄

Dongyu App session 只属于当前 workspace

一个浏览器可以同时登录多个 workspace，但 issuer、homeserver、数据权限和 session 要彼此独立。

禁止

Matrix 不转发普通登录 token

不要通过 Matrix 中转 ZITADEL access token / refresh token，也不要让中转 workspace 代持用户登录令牌。

官网账号

用于购买、文档、下载和 workspace discovery。它可以是中心化账号，但不能因为用户在官网登录了，就直接读取所有 workspace 数据。

  • 适合：公共门户、账单、帮助中心。
  • 不适合：直接授予 workspace 内数据权限。
  • 验证点：进入 App 前必须选择具体 workspace。

两种 federation 不要混用

ZITADEL 的 federation 是 IAM / SSO federation；Matrix 的 federation 是协议级跨 homeserver 通讯。它们可以配合，但不是同一种东西。

ZITADEL 适合做组织内 SSO 和 IdP brokering

它可以按 organization 配置不同外部 IdP，让用户通过 Entra ID、Okta、Google Workspace 等登录。

1用户进入当前 workspace 的 Dongyu App。
2App 跳转到该 workspace 选定的 ZITADEL / IdP。
3App 用返回的身份建立当前 workspace session。

Matrix 适合承担跨 workspace 管理总线

每个 workspace 可以有自己的 homeserver 或管理房间。跨 workspace 通讯由 Matrix transport 承载，但业务授权仍由目标 workspace 验证。

A源 workspace 生成短期 assertion。
BMatrix 管理总线转发请求。
C目标 workspace 依据 policy 决定是否执行。

MAS 是 Matrix 登录层，不是 Dongyu 通用 IAM

MAS 可以接 upstream OIDC provider，并服务 Matrix homeserver 登录。Dongyu App 可以通过 Matrix SSO connect 获取 Matrix 能力，但不应把 MAS 当成所有业务系统的权限中心。

1Dongyu App session 已经存在。
2用户需要 Matrix 能力时触发 Matrix SSO。
3后端换取 Matrix token 并仅保存在受控 session 中。

跨工作区请求流程

点击步骤查看它应校验什么。关键点是：中转只做目录、信任和策略辅助，不转发原始登录 token。

用户登录本工作区

用户只在 source workspace 建立本地 session。这个 session 不应自动变成访问其他 workspace 的通行证。

  • 校验：本地 IdP 与 workspace membership。
  • 输出：source workspace session。
  • 禁止：直接把官网登录态当成所有 workspace 权限。

短期 assertion / capability 字段检查器

点击字段查看它解决什么问题。字段越少越危险，尤其不能缺 audience、nonce、expiry 和签名。

iss

声明由哪个 workspace 或 policy broker 签发。目标 workspace 必须在 trust registry 中找到该 issuer 的公钥和允许范围。

  • 防止来源不明的请求被执行。
  • 必须与签名公钥、policy 版本和审计记录关联。
  • 失败处理：fail closed，并写可审计拒绝原因。

落地规则

这些规则可以直接拆进后续 plan / resolution 的验收项。

Token 规则 Matrix 中只传短期 assertion / capability，不传普通 access token / refresh token。 缺 signature / aud / exp 时拒绝。
Session 规则 同一浏览器可以有多个 workspace session，每个 session 独立绑定 issuer、homeserver 和数据权限。 官网登录不等于全部 workspace 已授权。
中转规则 中转 workspace 只做 trust registry、policy broker、公钥目录或 workspace 目录。 不能代持用户原始登录令牌。
目标侧规则 目标 workspace 自己校验 issuer、audience、nonce、expiry、policy 和 capability。 不能只因为来自 Matrix room 就执行。
