# Iteration 0372 Submit Button ModelTable Docs Plan

## Goal

- Extend the "最小 Submit 双总线示例" developer docs so submit-style buttons have a standalone, copyable ModelTable recipe: which labels are required, which labels participate in event dispatch, how the labels become effective, and how to extend the pattern to multiple submit buttons.

## Scope

- In scope:
- Update the Markdown guide.
- Update the interactive HTML guide.
- Add deterministic docs checks that lock the submit-button recipe.
- Record iteration evidence.
- Out of scope:
- Runtime behavior changes, JSON patch changes, zip contents, remote deployment, or local app redeploy.

## Invariants / Constraints

- UI remains projection-only; submit buttons do not directly write business result labels and do not directly call Matrix/MQTT.
- Button payloads remain temporary ModelTable records.
- Formal egress still goes through root pins, generated host adapter, and Model 0 management bus.
- Provider zip must not include host-owned labels such as `ui.egress.binding.v1`, `pin.bus.*`, generated cleanup labels, or `route.reply_to`.

## Success Criteria

- Markdown and HTML both explain:
  - button Cell labels;
  - `ui_bind_json.write.pin`, `value_t=modeltable`, and `value_ref`;
  - the temporary records generated by the click;
  - root `root_routes`, `submit_request`, `submit_request_wiring`, and `handle_submit`;
  - root `submit1` public egress and `dual_bus_model.egress_pins`;
  - the related status/result labels;
  - how to name multiple submit buttons.
- Existing Minimal Submit docs contract tests pass.
- Import/export docs tests pass.
- HTML renders in Playwright and the submit-button panel is reachable.
- `git diff --check` passes.
