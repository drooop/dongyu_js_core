# Iteration 0376-control-first-mbr-routing Plan

## Goal

Freeze and implement the control-bus-first routing contract for slide-app dual-bus traffic: UI Server must emit default slide-app traffic through control bus (`pin.bus.cb.*`) to MBR, and MBR must route by the ModelTable payload `topic` record rather than by endpoint metadata. MBR is refilled as Tier 2 table logic for the first two required duties: control bus to control bus, and control bus to management bus. MBR-to-MBR management messages stay out of scope for this iteration.

## Scope

- In scope:
- Update SSOT and user-guide documents so current truth says default same-workspace slide-app traffic is `UI Server -> control bus -> MBR -> control bus -> Remote Worker`.
- Define the emitted payload records needed for MBR routing:
  - Required record: `{ id: 0, p: 0, r: 0, c: 0, k: "topic", t: "str", v: "<fully expanded MQTT topic>" }`.
  - Optional record: `{ id: 0, p: 0, r: 0, c: 0, k: "route_kind", t: "str", v: "control" | "management" }`; missing `route_kind` means `"control"`.
  - The `topic` record is generated by UI Server / host-owned installer at runtime from provider stable route identity and current environment. Provider ZIP records must not be treated as a fully trusted transport topic unless the installer has validated and materialized the final `topic` record.
  - The `topic` record must be a safe MQTT topic: non-empty segments, no `+` or `#`, and no empty `//` segment.
  - Temporary ModelTable payload records remain the only business data format.
- Remove the old route truth from MBR forwarding: `endpoint_worker_id` / `endpoint_model_id` / `endpoint_pin` can remain recipient metadata for the remote worker runtime, but MBR must not use them to decide the MQTT topic.
- Update UI Server imported slide-app host egress generation so default host-owned egress uses `pin.bus.cb.out`, `bus="control"`, and includes the route topic/route kind records.
- Refill `deploy/sys-v1ns/mbr/patches/mbr_role_v0.json` so MBR handles `control -> control` and `control -> management` by Tier 2 functions.
- Update affected tests, minimal submit payload/docs, and local verification paths so no old `pin.bus.mb.*` default slide-app route remains.
- Run deterministic checks and real browser verification for Workspace slide-app install/run and the color generator path after local deployment/restart.
- Out of scope:
- MBR-to-MBR cross-workspace management bus delivery.
- Per-app static route registration in MBR.
- Compatibility parsing of old topic formats, old result topics, `return_topic`, `route.reply_to`, or endpoint-derived MBR routing.
- Redesigning Remote Worker internal model dispatch beyond what is needed to consume the same request topic and send response packets.

## Invariants / Constraints

- `CLAUDE.md` remains highest authority.
- Every business pin payload remains a Temporary ModelTable record array; no loose JSON object fallback.
- UI remains projection only; business events enter through Model 0 and explicit pin routes.
- No compatibility layer is allowed for old routing. If the required `topic` record is missing or invalid, or if `route_kind` is present but not `control` / `management`, the path must fail closed with visible error labels. Missing `route_kind` is valid and means `control`.
- Provider ZIP records must not declare `pin.bus.*` or host-owned bindings. Installation generates host-owned bus labels.
- MBR must route from payload `topic`, not from `endpoint_*` records.
- Same-workspace default route is control bus. Management bus is used only when explicitly requested or future cross-workspace rules require it.

## Success Criteria

- SSOT/user-guide state the new default control-bus-first path and distinguish MBR routing `topic` from remote worker endpoint metadata.
- Deterministic tests prove:
  - UI Server generated imported host egress defaults to `pin.bus.cb.out`.
  - Generated request payload contains `topic` and `route_kind="control"`.
  - MBR forwards `route_kind="control"` to `mbr_cb_out` using payload `topic`.
  - MBR forwards `route_kind="management"` to `mbr_mb_out`.
  - MBR accepts missing `route_kind` as `control`.
  - MBR rejects missing/unsafe topic and invalid `route_kind`.
  - MBR does not derive topic from endpoint records, `return_topic`, `route.reply_to`, topic suffixes, or result-topic conventions.
  - Old default slide-app `pin.bus.mb.out` assertions are removed or replaced.
- `mbr_role_v0.json` and the minimal submit payload/docs match the new contract.
- Local deployment/restart is performed before browser claims.
- Browser verification proves Workspace slide-app install/open/submit still works and color generator submit changes result state.
