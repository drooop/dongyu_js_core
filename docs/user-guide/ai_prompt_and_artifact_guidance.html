AI 协作规约与 Artifact 使用指南

本页把本项目的 AI 协作规则收成一张可读工作台：先分类规则，再选择表达方式；默认 Markdown / 文本，只有明确需要 visualized / interactive 文档或用户要求 HTML 时才使用 HTML。

使用结论

1

先定目标

说明这条规则要稳定什么行为，避免什么风险。

2

再分类

分成硬约束、判断规则、偏好建议，不把所有要求都写成绝对命令。

3

写验证

能运行就运行，能检查就检查；质量靠证据，不靠重复加重语气。

4

选格式

默认 Markdown / 文本；HTML 只用于明确的可视化、交互或显式要求。

三类规则

Invariant

硬约束

适用：安全禁区、数据真源、流程闸门、禁止操作、必填字段、语义合同。

写法：可以使用“必须 / 禁止 / 不得 / MUST / NEVER”。

例子：远端禁区命令不得执行；验证必须 PASS/FAIL。

Decision Rule

判断规则

适用：是否搜索、是否追问、是否继续探索、是否生成 artifact。

写法：条件 -> 动作 -> 停止条件 -> 验证。

例子：信息可能过期时查官方来源；上下文足够时直接推进。

Preference

偏好建议

适用：语气、篇幅、默认格式、协作节奏。

写法：默认倾向 + 让位条件。

例子：默认结论先行；用户明确要求时覆盖默认。

不推荐

回答前必须先搜索。
每次都生成 HTML artifact。
必须输出正确 JSON。

推荐

当信息可能变化或高风险时，先查权威来源。
默认使用 Markdown；只有明确需要 visualized / interactive 文档或用户要求 HTML 时才生成 HTML。
当输出要被程序读取时，提供 schema、字段表或例子，并验证可解析。

HTML / visualized artifact 边界

适合使用 HTML

  • 需要可视化比较多个方案。
  • 需要交互式筛选、展开、导出或演示。
  • 需要给非技术读者一个可打开的视觉说明页。
  • 用户明确要求 HTML。

不适合使用 HTML

  • 只是结论、短解释、普通修复汇报。
  • 只是为了显得完整或好看。
  • 会让 SSOT、iteration 记录和验证证据变得分散。

本项目里的 HTML artifact 是阅读和交互产物，不是 SSOT。真实规则仍以 CLAUDE.md、AGENTS.md、docs/ssot/、docs/WORKFLOW.md 和 docs/iterations/* 为准。

提交前检查清单

  • 规则是否已分成硬约束、判断规则、偏好建议。
  • 判断项是否写成条件规则，而不是无条件绝对命令。
  • 是否写了验证方式。
  • 是否避免把 HTML 变成默认输出。
  • 长期规则是否写进长期文档，而不是只留在聊天里。
