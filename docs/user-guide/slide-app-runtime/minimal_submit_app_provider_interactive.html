最小 Submit 双总线示例

这不是静态 HTML demo。它说明 Workspace 中的 cellwise UI 如何点击 Submit，经 Model 0、Matrix、MBR、MQTT 到 remote-worker R1，再把结果回流到 UI 模型。

UI model1050
Bus inbus_event_submit_1050_0_0_0
Submit topicUIPUT/ws/dam/pic/de/sw/1050/submit
Result topicUIPUT/ws/dam/pic/de/sw/1050/result

浏览器显示的是 remote-worker 回流结果

在 Workspace 打开 最小 Submit 双总线示例 后，输入内容并点击 Submit，结果应由 R1 处理后回写为 Submitted: <输入内容>。

最小 Submit 双总线示例
Waiting for submit
REMOTE idle

R1 只填 remote truth 和程序模型

      
新建 remote-worker R1 时，R1 上的模型 1050 不负责 UI 渲染；它只负责收 submit、跑程序、发 result。

      

        
model_typemodel.table
R1 的 1050 是可执行表模型。

        
submitpin.in
接收 UIPUT/ws/dam/pic/de/sw/1050/submit 上的 payload。

        
resultpin.out
程序处理完成后把结果发出。

        
root_routespin.connect.label
submit -> on_minimal_submit_matrix_remote_submit:in。

      

      
{
  "model_id": 1050,
  "cell": [0, 0, 0],
  "required_labels": [
    "model_type",
    "display_text",
    "remote_status",
    "submit_inflight",
    "last_submit_payload",
    "submit",
    "result",
    "result_out_topic",
    "root_routes",
    "on_minimal_submit_matrix_remote_submit"
  ]
}

      
R1 程序只接受 payload 中的 text record；不接受 input_value 旧字段兜底。

    


    

      
UI 是细粒度 cellwise 填表

      
页面不是 HTML 字符串。每个可见部件都是一个 cell，正式业务提交进入 Model 0。

      

        
(2,0,0)Container
页面根容器。

        
(2,1,0)Card
标题为 最小 Submit 双总线示例。

        
(2,3,0)Input
读写 input_text，commit_policy=on_blur。

        
(2,4,0)Button
bus_event_v2=true，提交临时 records。

        
(2,5,0)Text

(2,6,0)StatusBadge

用 Workspace 的“滑动 APP 导入”触发安装

      
如果交付新 APP，不直接改运行时。把模型表 records 打成 zip，再由 Workspace 导入器完成上传、materialize、mount 和自动引脚建立。

      

        
准备 zip
minimal-submit-dual-bus.zip 里只放 app_payload.json。

        
写 app_payload.json
内容是临时 ModelTable records array，使用 id: 0，不写正式 model id，不写 patch op。绑定里的 model_id: 0 或 *_model_id=0 都表示 zip 内根模型，会在导入时 remap。

        
或者导出 zip
如果 APP 已经在 Workspace 中填表完成，使用资产树里的 Zip 链接，或访问 /api/slide-apps/<modelId>/export.zip。

        
打开 Workspace
在资产树中打开 滑动 APP 导入。

        
选择 zip 并导入
导入器会调用 /api/media/upload，得到 mxc://...，再把 records materialize 成正式模型。

        
自动挂载
宿主把新模型挂到 Workspace，并按声明建立宿主侧引脚和 Model 0 adapter。

      

      
{
  "zip": "minimal-submit-dual-bus.zip",
  "files": ["app_payload.json"],
  "payload_shape": "ModelTable records array",
  "can_generate_by": ["write records", "Workspace Zip export"],
  "export_endpoint": "/api/slide-apps//export.zip",
  "model_id_remap": ["ui_bind_json.read.model_id", "ui_text_ref_model_id"],
  "do_not_include": ["op", "installed model_id", "secret", "token", "host_*_generated_*"]
}

    


    

      
外部客户端按 topic 收发

      
最直接的联调点是 MQTT topic。Matrix 客户端看到的是 dy.bus.v0 事件；普通第三方 Matrix 账号可能被 peer 过滤，联调时优先使用 MQTT。

      

        
subscribeUIPUT/ws/dam/pic/de/sw/1050/submit
观察浏览器点击 Submit 后 MBR 发给 R1 的消息。

        
publishUIPUT/ws/dam/pic/de/sw/1050/result
模拟 R1 回包，直接让 UI 显示更新。

      

      
{
  "version": "v1",
  "type": "pin_payload",
  "op_id": "manual_result_1050_001",
  "source_model_id": 1050,
  "pin": "result",
  "payload": [
    { "id": 0, "p": 0, "r": 0, "c": 0, "k": "display_text", "t": "str", "v": "Submitted: hello from external client" },
    { "id": 0, "p": 0, "r": 0, "c": 0, "k": "remote_status", "t": "str", "v": "remote_processed" },
    { "id": 0, "p": 0, "r": 0, "c": 0, "k": "submit_inflight", "t": "bool", "v": false }
  ],
  "timestamp": 1778115520131
}

      
这条消息发到 result topic 后，MBR 会转回 Matrix，ui-server 再 materialize 到 UI 模型 1050 的 display_text。

    


    

      
这个示例不保留兼容链路

      
如果这些写法出现在新示例中，应该直接修正，不做兼容。

      

        
pin.connect.model禁止
当前规约改用 pin.connect.cell 和 model.submt 边界。

        
ctx.writeLabel/getLabel/rmLabel禁止
程序模型写 label 使用当前 V1N 口径。

        
input_value fallback禁止
R1 submit 程序只读 text。

        
frontend direct Matrix禁止
正式业务先进入 Model 0。
