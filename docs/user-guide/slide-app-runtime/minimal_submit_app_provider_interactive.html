最小 Submit 双总线示例

这不是静态 HTML demo。它说明 Workspace 中的 cellwise UI 如何点击 Submit，经 Model 0、Matrix、MBR、MQTT 到 remote-worker RE，再按 server-owned reply_to 把结果回流到本地 UI 模型。

Local model2000
ProviderRE / 3000
Submit routeroute.to.pin=submit1
Reply routeserver-owned reply_to

浏览器显示的是 remote-worker 回流结果

在 Workspace 打开 最小 Submit 双总线示例 后，输入内容并点击 Submit，结果应由 RE 的 provider model 处理后回写为 Submitted: <输入内容>。

最小 Submit 双总线示例
Waiting for submit
REMOTE idle
UI local model 2000Button sends bus_event_v2; UI Server creates reply_to

RE 公开 pin，内部接程序模型

      
新建 remote-worker RE 时，RE 上的 provider model 3000 不负责 UI 渲染；它只负责公开 submit1 pin、跑程序、发 result。

      

        
model_typemodel.table
RE 的 3000 是 provider 表模型。

        
submit1pin.in
接收 route.to.pin=submit1 的 payload。

        
resultpin.out
程序处理完成后把结果发出。

        
submit1_routepin.connect.cell
[0,0,0,submit1] -> [1,1,1,submit1_in]。

        
submit1_wiringpin.connect.label
submit1_in -> submit1:in。

      

      
{
  "model_id": 3000,
  "cell": [0, 0, 0],
  "required_labels": [
    "model_type",
    "display_text",
    "remote_status",
    "submit_inflight",
    "last_submit_payload",
    "submit1",
    "result",
    "submit1_route",
    "submit1_wiring",
    "submit1"
  ]
}

      
RE 程序只接受 payload 中的 text record；不接受 input_value 旧字段兜底。函数端点不能跨 Cell 直连，必须先到普通 pin。

    


    

      
UI 是细粒度 cellwise 填表

      
页面不是 HTML 字符串。每个可见部件都是一个 cell；按钮点击先写入本 Cell 的 click_chain，再通过 root 接线触发 handle_submit，最后由 root submit1 pin.out 进入双总线。

      

        
(2,0,0)Container
页面根容器。

        
(2,1,0)Card
标题为 最小 Submit 双总线示例。

        
(2,2,0)Input
读写 input_text，commit_policy=on_blur。

        
(2,3,0)Button
ui_bind_json.write.pin=click_chain，提交临时 records。

        
(2,4,0)Text
读取 display_text。

        
(2,5,0)StatusBadge
读取 remote_status。

      

      
按钮事件绑定

      
ui_bind_json 的 value_ref 会被前端解析成 ModelTable-like records，并写到按钮 Cell 的 click_chain pin.in。这里不直接写 display_text，也不直接发 Matrix。

      
{
  "cell": [2, 3, 0],
  "labels": {
    "ui_component": "Button",
    "click_chain": "pin.in",
    "ui_bind_json.write.pin": "click_chain",
    "ui_bind_json.write.value_t": "modeltable",
    "ui_bind_json.write.value_ref": [
      "__mt_payload_kind=ui_event.v1",
      "text=$label(input_text)",
      "source=ui_button"
    ]
  }
}

      
从按钮到程序模型

      
root 的 root_routes 用 pin.connect.cell 把 [2,3,0,click_chain] 接到 root submit_request；root 的 submit_request_wiring 再用 pin.connect.label 触发 handle_submit:in。

      
{
  "root_labels": {
    "host_ingress_v1": "declares submit_request as import boundary",
    "remote_bus_endpoint_v1": "route.to worker/model = RE / 3000",
    "dual_bus_model": "egress_pins = [submit1]",
    "submit_request": "pin.in",
    "submit_request_wiring": "submit_request -> handle_submit:in",
    "root_routes": "[2,3,0,click_chain] -> [0,0,0,submit_request]",
    "handle_submit": "func.js, reads only text record",
    "submit1": "pin.out, public egress pin"
      }
}

      
完整触发链路

      
(2,3,0) Button ui_bind_json
-> writes temporary ModelTable records to (2,3,0) click_chain
-> root_routes: [2,3,0,"click_chain"] -> [0,0,0,"submit_request"]
-> submit_request_wiring: submit_request -> handle_submit:in
-> handle_submit func.js reads text and writes root submit1 pin.out
-> generated host egress adapter reads remote_bus_endpoint_v1 + dual_bus_model.egress_pins
-> Model 0 mt_bus_send / pin.bus.mb.out
-> Matrix dy.bus.v0 to @mbr:<host_url>
-> MBR derives MQTT topic from route.to = RE / 3000 / submit1
-> RE Model 3000 public submit1 pin
-> RE submit1 program returns pin_payload.v1 to route.reply_to
-> UI Server materializes display_text / remote_status on local installed model

      
当前示例不接受 input_value、message_text 或嵌套 value.text 兜底。payload 中没有 text 就按空输入处理。Submit 这类按钮只产生 ModelTable-like 事件；后续是否触发程序、是否外发双总线，由模型表引脚和接线决定。

      

    


    

      
Submit 类提交按钮如何准备模型表

      
提交按钮不是一个“直接调接口”的控件。它由按钮 Cell、root 入口、程序模型、root 外发 pin 和状态 labels 一起组成；点击时只产生临时 ModelTable records，正式业务再由模型表接线继续流动。

B. Root 入口与程序模型 labels

      
按钮 Cell 只写自己的 click_chain。要触发程序，root 必须声明 submit_request、root_routes、submit_request_wiring 和 handle_submit。

      

        
submit_requestpin.in
root 业务入口，接收按钮生成的 records。

        
root_routespin.connect.cell
[2,3,0,click_chain] -> [0,0,0,submit_request]。

        
submit_request_wiringpin.connect.label
submit_request -> handle_submit:in。

        
handle_submitfunc.js
读取 text，写状态，并写 submit1 pin.out。

      


      
C. 外发与回包相关 labels

      
需要双总线时，root 还要有 submit1 pin.out、dual_bus_model.egress_pins、remote_bus_endpoint_v1 和 host_ingress_v1。这些 labels 让安装器生成 host egress adapter；开发者 zip 不应手写 ui_egress_submit1_binding 或 pin.bus.mb.out。

      

        

          
label.klabel.t示例值如何生效

          
            
submit1pin.outnullhandle_submit 写入后进入宿主外发链路。

            
dual_bus_modeljsonegress_pins=["submit1"]安装器知道哪些 root pin 要接双总线。

            
remote_bus_endpoint_v1jsonRE / 3000安装器生成 route.to = RE / 3000 / submit1。

            
host_ingress_v1jsonpin_name=submit_request宿主知道如何从 Model 0 管理总线进入该 app。

          
        

      


      
D. 状态与结果 labels

      
input_text 被 value_ref 读取；submit_inflight、last_submit_payload、remote_status 和 display_text 让页面显示提交过程与远端结果。

      
input_text -> ui_bind_json.value_ref reads as text
handle_submit -> submit_inflight=true, remote_status=sending, last_submit_payload=...
remote-worker reply -> display_text="Submitted: ...", remote_status=remote_processed, submit_inflight=false


      
E. 生效顺序

      
Button Cell ui_component=Button makes the visual button
-> user clicks Button
-> ui_bind_json.write.value_t=modeltable creates temporary ModelTable records
-> frontend writes records to Button Cell click_chain pin.in
-> root_routes sends click_chain to root submit_request
-> submit_request_wiring calls handle_submit:in
-> handle_submit reads text and writes root submit1 pin.out
-> generated host egress adapter adds route.to and route.reply_to
-> Model 0 mt_bus_send / pin.bus.mb.out sends to Matrix / MBR / RE
-> RE reply materializes display_text and remote_status


      
多个提交按钮

      
多个按钮要成组命名。比如第二个按钮可以使用 approve_click_chain、approve_request、handle_approve 和 approve1，并把 dual_bus_model.egress_pins 扩展为 ["submit1","approve1"]。

      
不要复用同一个 click_chain 或 root submit_request，除非这些按钮确实要走完全相同的程序逻辑。

zip 进入 UI Server 后如何安装

      
UI Server 安装器会把 zip 中的临时 ModelTable records materialize 成正式模型，再把它挂成 Model 0 的子模型，并自动生成 host ingress / host egress labels。provider zip 不能自己写这些 host-owned labels。

      

        
解析和校验
zip 必须只有 app_payload.json；安装器拒绝 op、正式 model_id、route.reply_to、pin.bus.*、ui.egress.binding.v1 和 pin.connect.model。

        
分配正式模型 id
zip 内 id: 0 会映射成 UI Server 本地新模型，例如 1056；内部 model_id: 0、ui_text_ref_model_id=0 会同步 remap。

        
创建模型并写 provider labels
新模型类型为 sliding_ui，名称来自 app_name，zip 中合法 labels 被写入对应 cell。

        
写安装元信息
imported root 额外获得 deletable、installed_at、imported_bundle_model_ids、import_root_temp_id、from_user、to_user。

        
挂成 Model 0 子模型
Model 0 的 Workspace mount 区选择下一个空 cell，例如 (2,0,nextC)，写入 k=model_type, t=model.submt, v=<installedModelId>。

        
刷新侧边栏
导入程序调用 catalog refresh；侧边栏使用 app_name、source_worker、deletable、slide_capable、installed_at 和 Model 0 model.submt mount 显示新 App。

用 Workspace 的“滑动 APP 导入”触发安装

      
如果交付新 APP，不直接改运行时。把模型表 records 打成 zip，再由 Workspace 导入器完成上传、materialize、mount 和自动引脚建立。

      

        
准备 zip
minimal-submit-dual-bus.zip 里只放 app_payload.json。

        
写 app_payload.json
内容是临时 ModelTable records array，使用 id: 0，不写正式 model id，不写 patch op。绑定里的 model_id: 0 或 *_model_id=0 都表示 zip 内根模型，会在导入时 remap。

        
或者导出 zip
如果 APP 已经在 Workspace 中填表完成，使用资产树里的 Zip 链接，或访问 /api/slide-apps/<modelId>/export.zip。

        
打开 Workspace
在资产树中打开 滑动 APP 导入。

        
选择 zip 并导入
导入器会调用 /api/media/upload，得到 mxc://...，再把 records materialize 成正式模型。

        
自动挂载
宿主把新模型挂到 Workspace，并按声明建立宿主侧引脚和 Model 0 adapter。
