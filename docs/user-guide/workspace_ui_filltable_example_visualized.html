Workspace 文档页面填表示例 — 可视化指南

这份可视化文档现在对齐的是 0276 Doc Page Workspace Example，目标是展示：接近文档页面的界面，也可以通过填表组合出来。

1 总览：最终看到的页面

这不再是旧的三组件例子，而是正式的文档型 Workspace 页面。

2 模型关系：谁负责结构，谁负责内容

`1013` 负责结构，`1014` 负责文案和正文。

怎么看： `1013` 像页面骨架，决定有哪些块、顺序怎样排；`1014` 像正文来源，决定每一块写什么。

3 页面组成：每一块都来自填表

一个节点一个 cell，而不是整块页面一次性塞进去。

怎么看： 这张图直接对应 `Model 1013` 的 cell。你可以把它理解成“页面树”被拆成一格一格来填。

4 布局为什么真的是由 label 决定

不是换组件，也不是写死两套页面，只改一个标签就能变。

关键例子是：Model 1013 / (2,14,0) / ui_layout 
graph LR
    rowLabel["ui_layout = row"] --> rowView["左块 | 右块"]
    colLabel["ui_layout = column"] --> colView["左块\n右块"]

默认值

row。页面中“左侧块 / 右侧块”横向排列。

改表后

改成 column，同一个容器会变成上下排列。

怎么看： 同一个节点，不换组件，只改一个布局 label，页面排版就会改变。这就是“布局位置由 label 定义”的最直接证据。

5 内容为什么也是由 label 决定

正文内容主要来自 `Model 1014 / (0,0,0)`。

    intro["hero_intro"] --> heroP["Paragraph 节点"]

怎么看： 这页不是把整段长文塞进 host model。多数正文都在 `1014`，再通过绑定投射到 `1013` 的页面节点上。

6 最短重建思路

先有内容，再有结构，再有挂载。

flowchart TD
    S["开始"] --> A["创建 Model 1014\n填写内容 labels"]
    A --> B["创建 Model 1013\n填写节点结构 labels"]
    B --> C["在 1013 上挂 1014"]
    C --> D["在 Model 0 下挂 1013"]
    D --> E["写入 Workspace registry"]
    E --> F["刷新页面并打开"]
怎么看： 顺序不能反。先有 truth 内容，再有 host 结构，最后才是侧边栏入口。

7 你现在最该关注的点

  • 这页已经不是旧的三组件示例，而是正式的文档页面样例。
  • 结构主要在 `1013`，内容主要在 `1014`。
  • ui_layout / ui_order / ui_parent 决定“摆在哪里”。
  • doc_title / hero_intro / rebuild_step_* 决定“写什么”。

8 最短验证步骤

  1. 打开 Workspace。
  2. 找到 `0276 Doc Page Workspace Example`。
  3. 点击 `Open`，确认右侧出现文档页面。
  4. 去 Home，选择 `Model 1014`，修改 `doc_title` 或 `hero_heading`。
  5. 回 Workspace，确认标题或文案变化。
  6. 再去 Home，修改 `Model 1013 / (2,14,0) / ui_layout`。
  7. 回 Workspace，确认“左侧块 / 右侧块”从横排变成竖排，或反过来。
